# UX 与前端质量

把前端质量分为三个互相约束但不能互相替代的层面：

1. 产品 UX：用户能否理解、完成、撤销并从失败中恢复。
2. 视觉设计：界面是否根据真实主题和品牌形成明确、非模板化的表达。
3. 工程质量：实现是否可访问、响应式、稳定，并避免高影响性能问题。

## 权威顺序

按以下顺序解决冲突：

1. 用户当前明确要求和已确认产品决策。
2. `.rpi-outfile/specs/l0/ux-spec.md`、当前 Spec 和项目不变量。
3. 现有 Design System、品牌 Token 和标杆模块。
4. 本 Skill 的通用启发式。

不得用通用审美或性能建议覆盖项目已确认的用户流程。发现规范与代码冲突时报告差异，不静默改写规范。

## 按需加载参考

- 新建页面、视觉重设计或风格不明确时，读取 [references/visual-design.md](references/visual-design.md)。
- React/Next.js 新实现、重构或性能审查时，读取 [references/react-performance.md](references/react-performance.md)。
- 只做非 React 的局部 UX 检查时，不加载 React 参考。

## 执行流程

### 1. 建立上下文

读取当前任务、相关 Spec、UX 规范、标杆模块、现有组件、设计 Token、路由、数据契约和测试。确认：

- 目标用户、页面唯一主要任务和成功结果。
- 目标框架、渲染模式和浏览器/设备范围。
- 哪些视觉或交互要求已经确认，哪些仍是设计自由度。
- 本次是否改变既有流程、权限、资产可见性或失败恢复。

缺少视觉方向时可以提出并实现一个有依据的方向；缺少高影响产品决策时先停止并请求确认。

### 2. 先定义体验合同

编码前明确：

- 主流程、次要操作、返回路径和取消/撤销方式。
- loading、empty、success、error、partial、disabled 和 permission-denied 状态。
- 表单校验、提交中状态、防重复提交和失败后的输入保留。
- 删除、覆盖、发布、付费等高代价操作的确认与恢复。
- 键盘操作、焦点顺序、屏幕阅读器名称和减少动画偏好。
- 移动端窄屏、触摸目标、长文本、国际化数字/日期和内容溢出。

不得只实现默认成功状态。

### 3. 形成视觉方向

新设计先给出简短方案：主题依据、4–6 个颜色 Token、字体角色、布局概念和一个有理由的标志性元素。避免无依据套用常见 AI 模板、装饰性编号、随机渐变、过量圆角或分散动画。

大胆选择集中在一处；其他区域保持克制。结构、文案、动效和装饰都必须帮助理解内容或完成任务。

### 4. 实现并保持系统一致

- 优先复用现有组件和 Token；确有新语义时再扩展系统。
- 使用语义 HTML、明确标签、可见焦点和可预测的控件行为。
- 操作名称在按钮、确认、Toast 和结果状态中保持一致。
- 错误说明发生了什么以及用户下一步能做什么；空状态提供真实行动入口。
- 不以占位文字、颜色或图标作为唯一信息载体。
- 不为“看起来高级”牺牲可读性、性能或可访问性。

### 5. 应用框架相关性能规则

React/Next.js 优先检查：

1. 串行请求瀑布和可并行异步工作。
2. 重型依赖、barrel import、无条件加载和客户端 Bundle。
3. Server/Client 边界、重复序列化、认证和请求级缓存。
4. 不必要订阅、Effect 派生状态、组件内组件和高频重渲染。
5. 长列表、布局抖动、脚本加载、资源提示和 Hydration 闪烁。

先修 Critical/High 问题；没有测量或明确瓶颈时，不做可读性变差的微优化。

### 6. 验证真实界面

在条件允许时启动应用并检查真实渲染，而不只阅读 JSX/CSS：

- 至少验证桌面和移动视口。
- 通过键盘走完主要流程并检查焦点。
- 检查 loading、empty、error 和长内容状态。
- 尊重 `prefers-reduced-motion`。
- 检查控制台错误、布局溢出和明显的请求瀑布。
- 运行相关单元、组件、E2E、可访问性和性能检查。
- 支持截图时，用截图进行一次实现后自我批评，并移除一个无功能装饰或解释保留理由。

不能运行或渲染时，明确把结论标记为静态审查，不宣称视觉或运行行为已验证。

## 硬性失败项

- 表格 CRUD 使用同级常驻表单块，且项目规范未明确允许。
- 删除或不可逆操作没有确认、撤销或等价保护。
- 表单提交没有进行中状态、防重复提交或失败反馈。
- 交互控件不可通过键盘使用，或没有可见焦点。
- 图标按钮没有可访问名称。
- 动画忽略减少动画偏好并影响主要操作。
- 仅修改视觉外观，却破坏既有产品流程或 Design System。
- React/Next.js 存在已确认的高影响请求瀑布、认证缺失或大 Bundle 回归。
- 只有测试文件，没有真实测试或界面验证证据。

## 输出

设计任务先输出简短设计合同；审查任务输出按严重度排序的发现。每项发现包含：

```json
{
  "severity": "error | warning | suggestion",
  "category": "product-ux | visual-design | accessibility | responsive | react-performance | evidence",
  "file": "src/path/file.tsx",
  "line": 45,
  "message": "具体问题",
  "impact": "用户或运行影响",
  "recommendation": "最小可验证修复",
  "evidence": "规范、代码、截图或测试引用"
}
```

最终状态使用 `pass`、`fail` 或 `not_verified`。只有没有硬性失败项且完成要求范围内的真实验证时才能 `pass`。

## RPI 集成

前端任务关闭前，当 `frontend_ux_strict` 为 `true` 时执行：

```bash
/rpi-check ux
```

报告应关联当前 Task、Spec、修改路径和测试证据，供任务关闭与设计实现对账使用。
