# React 与 Next.js 性能审查

本参考改编自 Vercel Engineering `react-best-practices` Skill，固定参考版本 `f8a72b9603728bb92a217a879b7e62e43ad76c81`。源 Skill 声明采用 MIT License；许可证和归属说明见 [MIT.txt](MIT.txt)。本文件将原规则压缩为适合 RPI 设计实现对账的优先级检查表。

## 使用原则

- 先检查正确性、安全性和用户体验，再优化性能。
- 按 Critical → High → Medium → Low 排序，不以微优化替代架构问题。
- 对框架版本敏感的 API 先检查项目依赖和官方文档。
- 没有测量或明确证据时，不引入复杂缓存、Memo 或自定义状态层。

## Critical：消除请求瀑布

- 把只在分支中使用的 `await` 移入对应分支。
- 对互不依赖的请求使用并行启动和 `Promise.all`。
- API Route、Server Action 和 Loader 中尽早启动 Promise，靠近使用点再等待。
- 用组件组合和合理 Suspense 边界流式展示可先完成的内容。
- 避免父组件取完数据后子组件才开始独立请求。

验证：检查网络时间线、Server Trace 或代码依赖图，证明请求确实可并行。

## Critical：控制 Bundle

- 避免从大型 barrel 入口导入少量成员；优先使用可分析的直接路径。
- 重型编辑器、图表、地图和仅特定流程使用的模块按需动态加载。
- Analytics、客服、日志等非关键第三方脚本延后加载。
- 功能未启用时不要加载对应模块。
- 仅在 hover/focus 等真实用户意图出现时预加载昂贵资源。

验证：使用项目 Bundle Analyzer、构建输出或浏览器 Coverage 比较前后差异。

## High：Server 与 RSC 边界

- Server Action 与 API Route 一样执行认证、授权和输入验证。
- 使用请求级去重处理同一请求中的重复读取；跨请求缓存必须明确失效和容量。
- 不在模块级可变状态中保存请求、用户或租户数据。
- 缩小传给 Client Component 的序列化数据，避免重复传递同一大对象。
- 静态字体、图标或固定资源读取可提升到模块加载阶段，但不能包含请求态数据。
- 非关键日志或通知使用框架支持的响应后机制时，确认失败处理和运行环境兼容。

## Medium：客户端数据和事件

- 对重复客户端请求使用项目既有的数据层进行去重，不为单个页面引入第二套方案。
- 全局事件监听器集中注册、正确清理；滚动和触摸监听器在适用时使用 passive。
- localStorage/sessionStorage 数据保持最小、带版本，并处理解析失败与迁移。
- URL 应表达可分享、可返回的筛选、分页和导航状态。

## Medium：减少无价值重渲染

- 派生状态在 render 中计算，不用 Effect 同步第二份状态。
- 交互引起的逻辑放在事件处理器，不放入观察性 Effect。
- Effect 依赖使用稳定、最小的原始值。
- 高频瞬态且不影响渲染的值使用 ref。
- 昂贵初始值使用惰性 `useState` 初始化。
- 状态更新依赖旧值时使用函数式更新。
- 不在组件函数内部定义新的组件类型。
- 只在计算或子树确实昂贵且引用稳定有意义时使用 Memo。
- 非紧急更新使用 Transition/Deferred Value 前，先确认交互延迟来源。

## Medium：渲染和浏览器性能

- 长列表使用虚拟化或 `content-visibility`，并验证可访问性和滚动行为。
- 批量修改 DOM/CSS，避免读写布局交错导致 Layout Thrashing。
- 动画优先使用 transform/opacity；复杂 SVG 动画可移动到包装元素。
- 控制 SVG 精度和资源体积。
- 第三方脚本使用 `defer`/`async` 或框架脚本策略。
- Hydration 差异应通过一致数据源解决；只对已理解且无害的差异抑制警告。
- 资源提示只用于高置信关键资源，避免无效预连接和预加载。

## Low：JavaScript 微优化

只有 Profiler、Trace 或基准证明热点后才考虑：

- 为重复查找建立 Map/Set。
- 合并多次大数组遍历。
- 缓存循环内稳定属性、函数结果或 Storage 读取。
- 提前返回并先执行低成本条件。
- 对 min/max 使用单次循环而不是排序。
- 将稳定 RegExp 和静态 JSX 提升到组件外。

## 报告证据

性能发现必须说明：规则类别、受影响路径、用户影响、测量方式和最小修复。没有运行证据时标记为静态风险，不使用“已优化”结论。
