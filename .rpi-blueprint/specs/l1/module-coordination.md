# L1 模块：多模块协同规范

> 本模块为多模块项目提供系统化的协同规范，解决 AI 开发中常见的模块联动断裂、UX 流转不连贯问题。

## 适用场景

- 包含 ≥3 个功能模块的项目
- 存在跨模块数据联动的项目
- 需要统一业务流转的项目
- 多人协作的大型项目

## 不适用场景

- 单模块项目
- 模块间无联动的项目
- 简单的工具类项目

## 核心价值

1. **全局骨架优先**：在拆分模块前，先定义全局联动关系
2. **避免联动断裂**：通过明确的联动规则，避免 AI 遗漏联动逻辑
3. **统一技术实现**：确保所有模块使用统一的联动方式
4. **可追溯性**：记录完整的联动日志，便于问题定位

## 与 L0 的关系

## 开发流程增强

### 阶段 1：全局骨架定义

**时机**：项目初始化后，模块拆分前

**步骤**：
1. 使用 `/rpi-check skeleton-init` 初始化全局骨架
2. 定义所有模块的职责边界
3. 定义模块间的联动关系和数据流向
4. 前端项目定义跨模块 UX 流转规范
5. 使用 `/rpi-check skeleton` 验证骨架完整性

**输出**：
- `module-linkage.md`（模块联动关系）
- `ux-flow.md`（UX 流转规范，前端项目）

### 阶段 2：标杆模块开发

**时机**：全局骨架定义后，批量开发前

**步骤**：
1. 选择一个基础模块作为标杆（推荐用户管理或权限管理）
2. 严格按照 `module-linkage.md` 开发标杆模块
3. 实现标杆模块的联动逻辑（状态管理、事件触发、异常处理）
4. 完成后，整理标杆模块的联动实现到 `reference-module.md`
5. 后续所有模块必须参考标杆模块的联动方式

### 阶段 3：依赖顺序开发

**时机**：标杆模块完成后

**步骤**：
1. 按依赖顺序开发模块（先基础模块，再业务模块）
2. 使用 `/rpi-task start` 启动任务，自动注入联动上下文包
3. 在提示词中明确要求："参考标杆模块的联动实现，严格遵循 module-linkage.md"
4. 开发完成后，使用 `/rpi-check linkage` 检查联动完整性
5. 通过联动完整性门控后，才能关闭任务

### 阶段 4：端到端验证

**时机**：所有模块开发完成后

**步骤**：
1. 执行端到端业务流程测试
2. 验证跨模块联动是否正确
3. 验证 UX 流转是否连贯
4. 检查全局操作日志是否完整
5. 修复发现的问题并更新规范

## 质量门控增强

### 联动完整性门控

**配置位置**：`.claude/workflow/config/gates.json`

**配置示例**：
```json
{
  "verify": {
    "default": [
      {
        "name": "linkage_integrity_check",
        "command": "bash .claude/workflow/rpi.sh check linkage --quiet",
        "description": "检查跨模块联动是否完整实现"
      }
    ]
  }
}
```

**检查内容**：
- 模块联动关系是否与 `module-linkage.md` 一致
- 数据流向是否符合规范
- 是否存在联动遗漏或逻辑错误
- 异常处理是否完整
- 全局操作日志是否记录

### 自动检查脚本

**执行入口**：`bash .claude/workflow/rpi.sh check linkage --quiet`

**检查逻辑**：
1. 读取 `module-linkage.md` 中的联动关系表
2. 扫描所有模块的代码
3. 检查联动触发点是否实现
4. 检查数据传递是否符合规范
5. 输出联动遗漏和风险点

## 命令增强

### `/rpi-check skeleton-init`

**功能**：初始化多模块项目的全局骨架

**执行逻辑**：
1. 引导用户列出所有模块
2. 引导用户定义模块职责边界
3. 引导用户定义模块联动关系
4. 前端项目引导定义 UX 流转规范
5. 生成 `module-linkage.md` 和 `ux-flow.md`

**交互示例**：
```
请列出项目的所有模块（用逗号分隔）：
> 用户管理, 订单管理, 商品管理, 权限管理

模块 1: 用户管理
核心职责：
> 用户 CRUD、权限分配

唯一数据源：
> 用户信息（users 表）

禁止职责：
> 不得维护订单数据

依赖模块：
> 权限管理

...

是否为前端项目？(y/n)
> y

请定义核心业务流程（如：用户下单流程）：
> 1. 用户登录 -> 2. 浏览商品 -> 3. 创建订单 -> 4. 支付订单

...

✅ 全局骨架已生成：
- .rpi-outfile/specs/l0/module-linkage.md
- .rpi-outfile/specs/l0/ux-flow.md
```

### `/rpi-check skeleton`

**功能**：检查全局骨架完整性

**执行逻辑**：
1. 检查 `module-linkage.md` 是否存在
2. 检查模块职责边界表是否完整
3. 检查联动关系是否定义
4. 前端项目检查 `ux-flow.md` 是否存在
5. 输出缺失项和建议

**输出示例**：
```
全局骨架完整性检查：

✅ 通过 3 个检查：
- module-linkage.md 存在
- 模块职责边界表完整
- 联动关系已定义

❌ 发现 2 个问题：
1. 缺少 ux-flow.md（前端项目必需）
2. 订单管理模块的联动触发规则未定义

建议：
1. 使用 /rpi-check skeleton-init 补全 ux-flow.md
2. 在 module-linkage.md 中补充订单管理的联动触发规则
```

### `/rpi-check linkage`

**功能**：一键检查跨模块联动完整性

**执行逻辑**：
1. 读取 `module-linkage.md` 和全局 Spec
2. 扫描所有模块的联动实现
3. 对比联动关系表，检查是否有遗漏
4. 检查数据流向是否符合规范
5. 输出联动遗漏、逻辑错误、风险点

**输出示例**：
```
跨模块联动完整性检查：

❌ 发现 3 个问题：

1. 用户管理 -> 订单管理 联动遗漏
   规范：用户禁用时，必须关闭该用户未付款订单
   实际：未找到联动触发代码
   建议：在 src/modules/user/actions/disableUser.js 中添加事件触发

2. 订单管理 -> 商品管理 数据流向错误
   规范：订单创建时，通过事件总线通知商品模块扣减库存
   实际：直接调用了商品模块的内部方法
   建议：改用事件总线 eventBus.emit('order:created', {...})

3. 权限管理 -> 用户管理 异常处理缺失
   规范：权限变更失败时，必须记录日志并回滚
   实际：未找到异常处理代码
   建议：添加 try-catch 和回滚逻辑

✅ 通过 2 个检查：
- 全局操作日志记录完整
- 状态管理方式统一
```

## 最佳实践

### 1. 先全局骨架，再模块拆分

**错误做法**：
```
直接拆分模块 → 逐个开发 → 最后发现联动全错 → 大规模返工
```

**正确做法**：
```
1. 使用 /rpi-check skeleton-init 定义全局骨架
2. 明确所有模块的职责边界和联动关系
3. 开发标杆模块，验证联动方式
4. 按依赖顺序开发其他模块
5. 端到端验证业务流程
```

### 2. 按依赖顺序开发模块

**错误做法**：
```
随机顺序开发模块 → 依赖模块未完成 → 联动无法实现 → 等待或临时方案
```

**正确做法**：
```
1. 先开发基础模块（用户、权限等）
2. 再开发依赖基础模块的业务模块（订单、商品等）
3. 每完成一个模块，更新 reference-module.md
4. 后续模块直接参考已完成的模块
```

### 3. 使用统一的联动方式

**错误做法**：
```
A 模块用 Vuex → B 模块用 Pinia → C 模块直接调用 → 联动完全失效
```

**正确做法**：
```
1. 在 module-linkage.md 中明确技术实现标准
2. 标杆模块使用统一的状态管理和事件总线
3. 所有模块必须遵循相同的联动方式
4. 通过门控检查技术实现一致性
```

### 4. 记录完整的操作日志

**错误做法**：
```
联动失败 → 无日志 → 无法定位问题 → 盲目调试
```

**正确做法**：
```
1. 所有跨模块联动必须记录全局操作日志
2. 日志包含完整的上下文信息
3. 联动失败时，通过日志快速定位问题
4. 定期分析日志，优化联动逻辑
```

## 常见问题

### Q: 是否所有多模块项目都需要这个模块？

A: 不是。如果模块间无联动，或联动非常简单，可以不使用。主要适用于模块间有复杂联动关系的项目。

### Q: 标杆模块必须最先开发吗？

A: 是的。标杆模块是后续模块的参考模板，必须先开发并验证联动方式的正确性。

### Q: 联动规范定义后可以修改吗？

A: 可以。但修改后需要同步更新所有相关模块的联动实现，确保一致性。

### Q: 如何处理异步联动的失败？

A: 在 `module-linkage.md` 中定义补偿机制，联动失败时将任务加入补偿队列，定时重试。

### Q: 全局操作日志存储在哪里？

A: 在 `module-linkage.md` 中定义存储位置，推荐使用数据库表或专业的日志服务。

## 总结

本模块通过以下机制，解决 AI 开发中的多模块联动断裂问题：

1. **骨架优先**：强制在模块拆分前定义全局联动关系
2. **标杆模块**：通过参考模板统一联动方式
3. **自动注入**：确保 AI 始终能读取到联动规范
4. **自动检查**：通过门控和命令自动发现联动遗漏
5. **Hook 阻断**：在源头阻止不符合规范的实现
6. **操作日志**：记录完整的联动日志，便于问题定位
