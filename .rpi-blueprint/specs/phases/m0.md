# M0 Validate — 核心价值验证

> 产品不确定性仍然较高，但核心链路必须由 Spec + TDD 保护；比例只表达关注趋势。

## 阶段姿态

- 产品：用真实数据和真实操作验证一条核心用户链路。
- 工程：核心行为必须有 Spec、TDD 和必要的 AI Eval。
- 范围：优先减少功能，不以账户、后台或全平台支持制造伪完整性。
- 风险：隐私、支付、系统权限和不可逆操作仍执行最低治理标准。

## M0 入场检查清单

- [ ] discovery.md 已填写目标用户 + 核心场景
- [ ] M0 Must 功能 ≤ 3 条，每条可独立验收
- [ ] Won't 列表 ≥ 3 条（明确不做什么）
- [ ] tasks.md 存在 Task-001（第一个可执行任务）
- [ ] spec.md 已定义最小接口/数据结构
- [ ] 功能变更已有 CHG 记录，高影响 DEC 已解决
- [ ] 核心 Capability/Invariant 已建立或显式标记为候选

## M0 出场验收标准

- [ ] Task-001 通过 TDD Red→Green（有失败测试证据）
- [ ] 核心链路可端到端运行（手动或自动）
- [ ] 质量门控通过（`/rpi-gates run`）
- [ ] 无 `__REQUIRED__` 占位符残留
- [ ] 用户价值证据与工程通过证据均已记录
- [ ] 设计、代码、测试执行和迁移（如有）对账通过

## 典型 M0 反模式（需立即停止）

- 写了 5+ 个 Task 才动工
- spec.md 仍含大量"待确认"
- 开始实现 Won't 列表中的功能
- 跳过 Red 阶段直接写实现
