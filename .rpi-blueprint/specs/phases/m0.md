# M0 — 核心闭环验证

**Vibe:Spec = 6:4**（方向探索 > 规范固化）

## 比例含义

| 维度 | 6（Vibe） | 4（Spec） |
|------|-----------|-----------|
| 重心 | 跑通 1 条核心链路 | 锁住边界，不允许扩散 |
| 容忍 | 代码不完美、覆盖不全 | Must ≤ 3、Won't ≥ 3 |
| 禁止 | 过早抽象、过度配置 | 留空的 spec 字段 |

## M0 入场检查清单

- [ ] discovery.md 已填写目标用户 + 核心场景
- [ ] M0 Must 功能 ≤ 3 条，每条可独立验收
- [ ] Won't 列表 ≥ 3 条（明确不做什么）
- [ ] tasks.md 存在 Task-001（第一个可执行任务）
- [ ] spec.md 已定义最小接口/数据结构

## M0 出场验收标准

- [ ] Task-001 通过 TDD Red→Green（有失败测试证据）
- [ ] 核心链路可端到端运行（手动或自动）
- [ ] 质量门控通过（`/rpi-gates run`）
- [ ] 无 `__REQUIRED__` 占位符残留

## 典型 M0 反模式（需立即停止）

- 写了 5+ 个 Task 才动工
- spec.md 仍含大量"待确认"
- 开始实现 Won't 列表中的功能
- 跳过 Red 阶段直接写实现
