# M1 Stabilize — 产品稳定与集成验证

> 产品方向趋于稳定，重点固化契约、异常路径、集成和回归保护。

## 阶段姿态

- 产品：根据 M0 的真实反馈修正并稳定核心体验。
- 工程：固化 API、数据、权限和失败恢复契约。
- 范围：新 Must 必须来自验证证据和显式 Roadmap 决策，不能顺手扩张。
- 质量：建立回归、集成、AI Eval、基础可观测和可重复发布能力。

## M1 入场检查清单

- [ ] M0 所有 Task 已 closed（result=pass）
- [ ] spec.md 已补充接口契约（输入/输出/错误码）
- [ ] 集成测试骨架已建立（哪怕 0 通过率）
- [ ] M1 Task 已按用户结果垂直切片并写入 tasks.md
- [ ] Must 功能已在 M0 中完整跑通
- [ ] （前端项目）UX 收束：标杆模块已完成，UX 规范已固化，后续模块禁止偏离标杆

## UX 收束阶段（前端项目适用）

M1 是 UX 交互标准从"探索"转为"固化"的节点：

1. M0 阶段允许在标杆模块中试验交互方案
2. M1 入场时，标杆模块的交互模式即为项目标准
3. M1 阶段所有新模块必须对照标杆模块实现
4. 禁止在 M1 阶段重新设计已在标杆中确定的交互模式

检查命令：`/rpi-check ux`

## M1 出场验收标准

- [ ] 所有 Must 功能在集成环境可复现
- [ ] 契约测试（API/接口）通过
- [ ] 异常路径（错误处理、边界输入）有测试覆盖
- [ ] 质量门控通过（lint + typecheck + unit + integration）
- [ ] spec.md 与代码同步（无 stale 警告）

## 典型 M1 反模式（需立即停止）

- 未经用户反馈和 Roadmap 决策直接新增 Must
- 集成测试全部跳过进入 M2
- spec.md 未更新契约就关闭任务
- 修改核心数据结构而未更新相关 spec
