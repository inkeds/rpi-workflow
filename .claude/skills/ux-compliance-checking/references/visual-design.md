# 有依据的视觉设计

本参考用于新页面、视觉重设计或现有界面缺少明确视觉方向时。它改编自 Anthropic `frontend-design` Skill，固定参考版本 `9d2f1ae187231d8199c64b5b762e1bdf2244733d`，并结合 RPI 的事实边界、Spec 和验证流程。原作采用 Apache License 2.0；许可证见 [APACHE-2.0.txt](APACHE-2.0.txt)。本文件已做结构、措辞和流程修改。

## 从主题而不是模板出发

先明确一个具体主题、目标用户和页面唯一任务。视觉线索应来自产品自己的材料、工具、对象、环境和语言，而不是来自通用 SaaS 模板。

如果用户已给出视觉方向，严格遵循。仍有自由度时，不要默认使用常见 AI 风格，例如：暖米色加高对比衬线体、近黑背景加单一荧光强调色、报纸式细线密集网格、无依据的玻璃拟态和随机渐变。

## 设计方案

编码前确定：

- Color：4–6 个有名称的颜色 Token 及使用角色。
- Type：展示、正文和必要的工具/数据字体角色。
- Layout：内容层级、密度、节奏和响应式变化。
- Signature：一个与主题相关、可解释的标志性元素。
- Motion：只选择一个主要动效时刻，或明确不使用动画。

用简短文字或 ASCII 线框比较布局。再次检查每项选择是否也会被不加修改地用于另一个无关产品；如果会，重新设计。

## 视觉原则

- Hero 是页面主张，不是“大标题 + 小标签 + 三个数字”的固定公式。
- 字体承担性格和层级，不只是装载文字；控制字体数量并保证可读性。
- 编号、标签、分隔线和装饰必须表达真实结构。内容不是序列时不要使用 01/02/03。
- 极简设计依赖精确间距、排版和细节；极繁设计需要足够的执行完整度。复杂度必须匹配方向。
- 把大胆选择集中在标志性元素，其余区域保持安静。
- 动效用于解释状态、空间关系或操作结果。零散 hover 特效通常不构成视觉方向。

## 文案也是设计材料

- 从用户可识别和可控制的对象命名，不暴露内部实现术语。
- 使用主动、具体的操作名称，例如“保存更改”，而不是“提交”。
- 同一操作在按钮、确认、Toast 和结果页中使用同一动词。
- 错误说明问题和恢复方式；空状态邀请用户采取下一步行动。
- 标签负责命名，示例负责演示，不让一个元素偷偷承担多个职责。

## 实现后批评

查看真实页面或截图并回答：

1. 最令人记住的元素是否确实来自主题？
2. 去掉品牌名称后，它是否仍像任何通用模板？
3. 是否有装饰没有帮助理解或操作？
4. 字体、间距、焦点、移动端和减少动画是否达到质量底线？
5. CSS 选择器、组件覆盖或响应式规则是否互相抵消？

移除一个无功能装饰，或明确记录保留它的产品理由。
