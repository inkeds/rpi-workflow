# UX 与通用前端质量

把前端质量分为三个互相约束但不能互相替代的层面：

1. 产品 UX：用户能否理解、完成、撤销并从失败中恢复。
2. 视觉设计：界面是否根据真实主题和品牌形成明确、非模板化的表达。
3. 工程质量：实现是否可访问、响应式、稳定，并避免高影响性能问题。

## 权威顺序

按以下顺序解决冲突：

1. 用户当前明确要求和已确认产品决策。
2. `.rpi-outfile/specs/l0/ux-spec.md`、当前 Spec 和项目不变量。
3. 现有 Design System、品牌 Token 和标杆模块。
4. 本 Skill 的通用启发式。

不得用通用审美或性能建议覆盖项目已确认的用户流程。发现规范与代码冲突时报告差异，不静默改写规范。

## 执行流程

### 1. 建立上下文

读取当前任务、Change/Decision/Conflict、相关 Spec、UX 规范、标杆模块、现有组件、设计 Token、路由、数据契约和测试。确认：

- 目标用户、页面唯一主要任务和成功结果。
- 技术栈、渲染模式和浏览器/设备范围。
- 哪些视觉或交互要求已经确认，哪些仍有设计自由度。
- 本次是否改变既有流程、权限、资产可见性、计费或失败恢复。

缺少视觉方向时可以提出并实现一个有依据的方向；缺少高影响产品决策或存在 pending `CNF-*` 时停止并请求裁决。

### 2. 先定义体验合同

编码前明确：

- 主流程、次要操作、返回路径和取消/撤销方式。
- loading、empty、success、error、partial、disabled 和 permission-denied 状态。
- 表单校验、提交中状态、防重复提交和失败后的输入保留。
- 删除、覆盖、发布、付费等高代价操作的确认与恢复。
- 键盘操作、焦点顺序、屏幕阅读器名称和减少动画偏好。
- 移动端窄屏、触摸目标、长文本、国际化数字/日期和内容溢出。

不得只实现默认成功状态。

## 有依据的视觉设计

### 从主题而不是模板出发

先明确具体主题、目标用户和页面唯一任务。视觉线索应来自产品自己的材料、工具、对象、环境和语言，而不是通用 SaaS 模板。

用户已给出视觉方向时严格遵循。仍有自由度时，不默认使用常见 AI 风格，例如无依据的玻璃拟态、随机渐变、装饰性编号、近黑背景加单一荧光色，或可套用任何产品的卡片网格。

### 编码前形成设计方案

- Color：定义 4–6 个有名称的颜色 Token 及使用角色。
- Type：定义展示、正文和必要的数据/工具字体角色。
- Layout：定义层级、密度、节奏和响应式变化。
- Signature：选择一个与主题相关、可解释的标志性元素。
- Motion：只选择一个主要动效时刻，或明确不使用动画。

用简短文字、ASCII 线框或已有页面比较布局。再次检查每项选择是否会不加修改地用于另一个无关产品；如果会，重新设计。

### 视觉与文案原则

- Hero 是页面主张，不是固定的“大标题 + 小标签 + 三个数字”。
- 字体承担性格和层级；控制字体数量并保证可读性。
- 编号、标签、分隔线和装饰必须表达真实结构。
- 极简依赖精确间距、排版和细节；极繁需要足够的执行完整度。
- 把大胆选择集中在标志性元素，其余区域保持安静。
- 动效用于解释状态、空间关系或操作结果，不用零散 hover 冒充视觉方向。
- 使用用户可识别和可控制的对象命名，不暴露内部实现术语。
- 同一操作在按钮、确认、Toast 和结果页中使用同一动词。
- 错误说明问题和恢复方式；空状态提供真实行动入口。

### 实现后视觉批评

查看真实页面或截图并回答：

1. 最令人记住的元素是否来自产品主题？
2. 去掉品牌名称后，它是否仍像任何通用模板？
3. 是否有装饰没有帮助理解或操作？
4. 字体、间距、焦点、移动端和减少动画是否达到底线？
5. CSS、组件覆盖或响应式规则是否互相抵消？

移除一个无功能装饰，或记录保留它的产品理由。

## 实现与系统一致性

- 优先复用现有组件和 Token；确有新语义时再扩展系统。
- 使用语义 HTML、明确标签、可见焦点和可预测的控件行为。
- 错误、空状态和权限不足必须提供下一步，而不只显示状态。
- 不以占位文字、颜色或图标作为唯一信息载体。
- 不为“看起来高级”牺牲可读性、性能或可访问性。

## 通用前端性能

先检查正确性、安全性和用户体验，再按 Critical → High → Medium → Low 优化。框架版本敏感的 API 必须对照项目依赖和官方文档；没有测量或明确证据时，不引入复杂缓存、Memo 或第二套状态层。

### Critical：消除请求瀑布

- 只在分支使用的数据，将等待动作移入对应分支。
- 对互不依赖的请求并行启动，在真正消费结果时等待。
- Route Handler、Loader、Server Action、服务端页面和客户端数据层都检查串行依赖。
- 使用框架支持的流式渲染、异步边界或占位状态，让可先完成的内容先展示。
- 避免父视图完成全部取数后，子视图才开始独立请求。

验证网络时间线、Server Trace 或依赖图，证明请求确实可以并行。

### Critical：控制 Bundle 和加载边界

- 避免从大型聚合入口只导入少量成员；优先使用可静态分析的直接路径。
- 编辑器、图表、地图和仅特定流程使用的模块按需加载。
- Analytics、客服、日志等非关键第三方脚本延后加载。
- 功能未启用时不加载对应模块。
- 只在 hover、focus 或可预测导航等真实意图出现时预加载昂贵资源。

使用 Bundle Analyzer、构建输出或浏览器 Coverage 比较前后差异。

### High：服务端、客户端和缓存边界

- 所有服务端入口都执行认证、授权和输入验证。
- 同一请求内的重复读取可以去重；跨请求缓存必须说明失效、隔离和容量。
- 不在模块级可变状态中保存请求、用户或租户数据。
- 缩小服务端传给客户端的序列化数据，避免重复传递大对象。
- 静态资源可提升到模块加载阶段，请求态数据不可以。
- 响应后日志或通知必须确认失败处理和运行环境兼容性。

### Medium：客户端数据、状态和事件

- 重复客户端请求使用项目既有数据层去重，不为单页引入第二套方案。
- 全局事件监听集中注册并正确清理；滚动和触摸监听在适用时使用 passive。
- Storage 数据保持最小、带版本，并处理解析失败与迁移。
- URL 表达可分享、可返回的筛选、分页和导航状态。
- 派生数据优先从权威状态计算，不维护容易失同步的第二份状态。
- 交互逻辑放在事件路径，观察/副作用机制只处理真正的外部同步。
- 高频瞬态且不影响渲染的值不要进入响应式渲染状态。
- 只在计算、依赖追踪或子树确实昂贵时使用缓存和记忆化。

### Medium：渲染与浏览器性能

- 长列表使用虚拟化、分页或 `content-visibility`，并验证可访问性和滚动行为。
- 批量修改 DOM/CSS，避免布局读写交错造成 Layout Thrashing。
- 动画优先使用 transform/opacity；复杂 SVG 动画可移动到包装元素。
- 控制图片、SVG、字体和第三方脚本体积。
- SSR/Hydration 差异通过一致数据源解决，不随意抑制警告。
- 资源提示只用于高置信关键资源，避免无效预连接和预加载。

### Low：有证据后再做微优化

只有 Profiler、Trace 或基准证明热点后才考虑：

- 为重复查找建立 Map/Set。
- 合并多次大数组遍历。
- 缓存循环内稳定属性、函数结果或 Storage 读取。
- 提前返回并先执行低成本条件。
- 对 min/max 使用单次循环而不是排序。
- 将稳定正则、常量模板和静态节点提升到重复执行路径之外。

## 框架映射

通用原则保持不变，框架机制按项目选择：

| 问题 | React / Next.js | Vue / Nuxt | Svelte / SvelteKit | Angular | 原生 Web / Astro |
|---|---|---|---|---|---|
| 派生状态 | render、selector；避免 Effect 复制状态 | `computed`；谨慎 `watch` 复制状态 | `$derived`/响应式声明 | signal/computed、RxJS selector | 直接计算或单一 store |
| 无效更新 | 订阅边界、稳定 props，必要时 memo | 组件边界、`computed`、`shallowRef` | 精确响应式依赖 | OnPush、signals、trackBy | 限制 DOM 更新范围 |
| 异步取数 | Loader/RSC/Route Handler、Suspense | `useAsyncData`、server route、Suspense | `load`、server route、await block | resolver、service、SSR | server endpoint、island data |
| 按需加载 | `dynamic` / `import()` | `defineAsyncComponent` / `import()` | `import()` / 路由拆分 | lazy route / `import()` | `import()` / island hydration |
| SSR 边界 | Server/Client Component、Hydration | server/client plugin、Hydration | server/browser boundary | SSR/Hydration | 静态优先、按需 island |
| 长列表 | 项目虚拟化组件 | 项目虚拟化组件 | 虚拟化或分页 | CDK virtual scroll | 虚拟化、分页、`content-visibility` |

框架专属规则只能在对应技术栈使用：React 的 Hook、Next.js Server/Client Component、Vue 的 `watch/computed`、Angular Change Detection 等不能互相机械套用。

## 验证真实界面

在条件允许时启动应用并检查真实渲染，而不只阅读模板或组件代码：

- 至少验证桌面和移动视口。
- 通过键盘走完主要流程并检查焦点。
- 检查 loading、empty、error、权限和长内容状态。
- 尊重 `prefers-reduced-motion`。
- 检查控制台错误、布局溢出、请求瀑布和 Bundle 变化。
- 运行相关单元、组件、E2E、可访问性和性能检查。
- 支持截图时，用截图进行一次实现后自我批评。

不能运行或渲染时，把结论标记为静态审查，不宣称视觉或运行行为已验证。

## 硬性失败项

- 不可逆操作没有确认、撤销或等价保护。
- 表单提交没有进行中状态、防重复提交或失败反馈。
- 交互控件不可通过键盘使用，或没有可见焦点/可访问名称。
- 动画忽略减少动画偏好并影响主要操作。
- 只修改视觉外观，却破坏既有产品流程或 Design System。
- 存在已确认的高影响请求瀑布、认证缺失或 Bundle 回归。
- 只有测试文件，没有真实测试或界面验证证据。

## 输出

设计任务先输出简短设计合同；审查任务输出按严重度排序的发现。每项发现包含：

```json
{
  "severity": "error | warning | suggestion",
  "category": "product-ux | visual-design | accessibility | responsive | frontend-performance | evidence",
  "file": "src/path/file",
  "line": 45,
  "message": "具体问题",
  "impact": "用户或运行影响",
  "recommendation": "最小可验证修复",
  "evidence": "规范、代码、截图、Trace 或测试引用"
}
```

最终状态使用 `pass`、`fail` 或 `not_verified`。只有没有硬性失败项且完成要求范围内的真实验证时才能 `pass`。

## RPI 集成

前端任务关闭前，当 `frontend_ux_strict` 为 `true` 时执行：

```bash
/rpi-check ux
```

报告关联当前 Task、Change/Conflict、Spec、修改路径和测试证据，供设计实现对账使用。
