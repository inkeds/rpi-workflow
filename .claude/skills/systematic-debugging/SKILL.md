# 系统化调试

本 Skill 改编自 `obra/superpowers` 的 systematic-debugging 与 test-driven-development，固定参考版本 `d884ae04edebef577e82ff7c4e143debd0bbec99`，并改写为 RPI 的 Task、Spec、Change 和 Evidence 流程。许可证见 [references/MIT.txt](references/MIT.txt)。

## 规则

没有根因调查，不提出修复。一次只测试一个假设，不把多个猜测堆进同一补丁。

## 1. 复现和收集证据

- 完整阅读错误、堆栈、路径、退出码和警告。
- 写出最小复现步骤，确认是否稳定发生。
- 检查近期 diff、依赖、配置、环境和数据变化。
- 对多组件链路记录每个边界的输入、输出、身份、配置和状态，先定位在哪一层断裂。
- 从错误位置向上追踪坏值或错误状态的来源，修复源头而不是最终症状。

不能复现时增加诊断证据，不猜测修复。

## 2. 对比模式

寻找同仓库中相似且工作的实现。完整比较合同、调用顺序、环境、生命周期和依赖，列出差异，不提前排除“小差异”。

同时判断问题属于：

- Spec 缺失或过期。
- 实现偏差。
- 测试/夹具错误。
- 环境或门禁基础设施问题。
- 新需求或产品决策变化。

## 3. 单一假设实验

明确写出：“根因可能是 X，因为证据 Y”。用最小、可逆变化验证一个变量。失败后撤销实验，结合新证据形成新假设，不在失败补丁上继续叠加。

连续三次修复假设失败时停止，检查是否存在架构耦合、共享状态或错误边界设计，并向用户报告，不直接尝试第四个补丁。

## 4. 回归测试和修复

先建立能复现原问题的失败测试或脚本，观察正确失败。随后只修复根因，运行：

1. 原始复现。
2. 新回归测试。
3. 相关测试范围。
4. 受影响的 Gate、联动或 Reconciliation。

## 5. 保存调试记录

在当前 Task 的 debug 证据中记录复现命令、观察、根因假设、被否定假设、最终修复和验证。若根因是需求变化，停止代码修补并进入 Change/Decision/Spec 流程。
