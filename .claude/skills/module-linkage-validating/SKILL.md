# 验证模块联动

## 1. 建立联动地图

读取 `.rpi-outfile/specs/l0/module-linkage.md`、可选 `ux-flow.md`、当前 Spec、接口/事件 Schema、迁移和相关测试。列出每条联动的生产者、消费者、合同、方向、所有者和失败处理。

项目规范优先于通用的事件命名或通信模式；不得为了统一形式擅自把 API、事件或共享状态互相替换。

## 2. 检查合同与所有权

- 每个已声明联动都有实现，每个新增联动都有设计声明。
- 跨模块只通过公开合同通信，不依赖对方内部实现。
- 数据所有者唯一；跨域写入、缓存和派生数据责任明确。
- API、事件、队列和持久化 Schema 的版本及兼容策略一致。
- 身份、租户、权限、幂等键和 Trace Context 能跨边界正确传播。

## 3. 检查失败语义

- 超时、重试、重复、乱序、部分成功和消费者不可用有明确行为。
- 不可原子完成的工作定义补偿、回滚或人工恢复。
- 重试不会重复扣费、重复创建资产或破坏账本一致性。
- 日志和指标能够定位失败发生在哪个边界，但不泄露秘密或敏感内容。

## 4. 验证端到端路径

至少覆盖成功、上游失败、下游失败、权限拒绝和恢复/重试路径。优先运行真实合同、集成或 E2E 测试；仅 Mock 单个模块不能证明联动成立。

```bash
/rpi-check linkage
```

## 5. 输出

报告模块、联动、合同版本、测试命令和证据。发现隐式联动、未知所有权、无恢复路径或只有静态推断时返回 `fail` 或 `not_verified`，不能返回 pass。
